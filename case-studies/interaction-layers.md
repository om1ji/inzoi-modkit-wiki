# Слои интеракции: как собран один пункт меню (Toilet01)

Интерактивная схема семи слоёв, из которых собран любой пункт меню объекта — на живом примере
унитаза `Toilet01`. Сплошные узлы вытащены из редактора через MCP; пунктирная цепочка `Toilet01_Bonus`
— предложение нового третьего пункта меню («+50 Mew»), ещё не записанное в игру.

## Семь слоёв сверху вниз

Каждый слой ссылается на следующий по строковому ID — это не одна сквозная таблица, а цепочка
независимых строк в разных таблицах. Слои 0–1 — как предмет доходит до своего реестра меню; слои
2–6 — сам конвейер интеракции.

| Слой | Таблица | Роль |
|---|---|---|
| **0 · Предмет** | `Object` | общие свойства предмета (цена, теги, налог); `ObjectTemplateId` → шаблон |
| **1 · Шаблон** | `ObjectTemplate` | привязка к blueprint-актору; `ObjectSelectionSetId` — мост к реестру меню |
| **2 · Реестр меню** | `ObjectSelectionSet` | какие пункты доступны у предмета в каждом состоянии (`Normal`/`Use`/`Broken`). `InteractionBagList` группы `Use` = меню, которое видит игрок |
| **3 · Пункт меню** | `InteractionBag` | название, награда, ветвление (`InteractionSequence`), **точка подключения скрипта №1** (`FinishScriptIds`) |
| **4 · Обёртка** | `Interaction` | фильтр цели, слот-теги, список `Action`'ов |
| **5 · Действие** | `Action_Interaction` | физическое действие: анимация (`Command: PlayAnim`) + **точка подключения скрипта №2** (`FinishScriptIdList` на шаг) |
| **6 · Эффект** | `Script_Interaction` | что происходит: `Command: Water_Usage` (ваниль) или `AddCurrency` (наши деньги) |

## Как эта цепочка читается словами

Проще всего понять устройство, если проследить его сверху вниз как единый рассказ, обращая внимание
на то, **какое поле в какую строку ведёт**.

Всё начинается с предмета — строки `Object.Toilet01`. Она хранит цену, теги и налог, но про меню и
действия не знает ничего. У неё есть только одна «ниточка» наружу — поле `ObjectTemplateId`, и эта
ниточка ведёт к строке `Toilet01_Template`.

`ObjectTemplate` описывает предмет уже как объект на сцене (меши, физика, намокание). И снова — ради
меню важно ровно одно его поле, `ObjectSelectionSetId`. Оно тянется к строке `ObjectSelectionSet.Toilet01`.
Получается, что «какие действия доступны» подключены к предмету не напрямую, а через шаблон:
`Object` → `ObjectTemplate` → `ObjectSelectionSet`.

`ObjectSelectionSet` — первое место, где реально решается, что можно сделать с унитазом. Строка
разбита по состояниям (`Normal` / `Use` / `Broken`), и в каждом лежит `InteractionBagList` — простой
список ID пунктов меню. В состоянии `Use` там два ID: `Toilet01_Use` и `Toilet01_Use_Flush`. Каждый
ID — это ссылка вниз, на строку `InteractionBag` с тем же именем.

`InteractionBag` — это уже сам пункт меню со всей начинкой: награда, ветвление, текст названия
(через `DisplayInfo.DisplayTextId` — ссылку в локализацию). Но действие он выполняет не сам: внутри
`InteractionSequence` есть поле `InteractionId`, которое указывает на строку `Interaction`. Так пункт
меню передаёт управление ниже.

`Interaction` готовит действие — к какому слоту подойти, какой фильтр цели — и через `ActionIdList`
перечисляет строки `Action`, которые надо выполнить по порядку (например, сначала `MoveRotate`,
потом основное действие). А `Action` — самый нижний слой: команда `PlayAnim` проигрывает анимацию, и
у неё же есть поле для скриптов-эффектов.

И на самом дне — `Script_Interaction`: строка с командами вроде `AddCurrency` или `Water_Usage`.
Сама она не знает, кто её вызвал; её «подтягивает» одно из полей-ссылок выше (`FinishScriptIds` у
пункта меню или `FinishScriptIdList` у шага действия). Об этих двух точках — ниже.

## Две независимые точки подключения скрипта-эффекта

Ключевой момент, который легко пропустить: `Script_Interaction` подключается **не** в одном месте
сквозной цепочки, а в двух параллельных, независимых точках. На схеме это видно на слое 6 — два
скрипта приходят из разных слоёв:

| Точка | Поле | Когда срабатывает | Пример на схеме |
|---|---|---|---|
| **№1 — уровень пункта** | `InteractionBag.FinishScriptIds` | один раз по завершении **всего** действия | наши деньги `AddCurrency_Toilet_Bonus` |
| **№2 — уровень шага** | `Action.Actions[].FinishScriptIdList` | привязан к конкретной **анимации/шагу** внутри `Interaction → Action` | ванильное списание воды `Water_Usage_10` |

Для нового пункта «деньги» проще точка №1 — `Interaction`/`Action` при этом вообще можно не трогать
(переиспользовать существующие). Точка №2 нужна, только если эффект должен сработать именно на
конкретном шаге анимации.

## Куда встаёт третий пункт

У `Toilet01` в группе состояния `Use` сейчас ровно две записи — `Toilet01_Use` («Использовать») и
`Toilet01_Use_Flush` («Слить воду»). Третий пункт добавляется так:

1. **Скопировать** `Toilet01_Use_Flush` в таблицах `InteractionBag` / `Interaction` /
   `Action_Interaction` под новым ID (`Toilet01_Bonus`) — так все обязательные поля уже заполнены
   корректной структурой.
2. **Создать** `Script_Interaction.AddCurrency_Toilet_Bonus` с командой `AddCurrency` (`S1: Mew`,
   `F1: 50`).
3. **Подключить** его в `InteractionBag.Toilet01_Bonus.FinishScriptIds`.
4. **Дописать** `"Toilet01_Bonus"` в `ObjectSelectionSet.Toilet01` → группа `Use` →
   `InteractionBagList` (не заменяя существующие).

```{caution}
Все правки — **вручную в редакторе ModKit**, не через `set_data`/`set_data_batch`: они стрингифицируют
массивы и структуры (`InteractionSequence`, `ActionIdList`, `InteractionBagList`, `Scripts`). См.
{doc}`../architecture/13-gotchas`. BUILD-воркспейс для `Toilet01` создавать не нужно — правки чисто
табличные, а нетронутый BUILD-воркспейс ловит баг с исчезновением 3D-модели.
```
