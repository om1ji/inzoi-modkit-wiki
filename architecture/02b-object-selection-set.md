# Что такое ObjectSelectionSet

`ObjectSelectionSet` — это **реестр меню предмета**: строка таблицы, которая отвечает на вопрос «какие
пункты доступны у этой вещи и в каком её состоянии». Это точка входа во всю систему интеракций: когда
игрок кликает по предмету, игра идёт от предмета к его шаблону, а от шаблона — сюда, чтобы собрать
список доступных действий.

```
Object → ObjectTemplate → (ObjectSelectionSetId) → ObjectSelectionSet → [InteractionBag, InteractionBag, …]
```

Ключевая идея: сам по себе `ObjectSelectionSet` **не описывает действия** — он лишь ссылается на них по
ID. Вся начинка действия (награда, анимация, эффект) живёт в `InteractionBag` и ниже
(см. {doc}`05a-interaction-bag`). Реестр только группирует ссылки по состоянию предмета.

## Поля верхнего уровня

Тип строки в движке — `FB1ObjectSelectionSetTableRow`. Описания полей — из официального
[StructGuide](https://mod-docs.playinzoi.com/Wiki/StructGuide/ObjectSelectionSet.html), значения — из
живых данных.

- **`Id`** (`FName`) — уникальный ID реестра. Обычно совпадает с именем предмета (`Toilet01`,
  `Computer01`, `Bed01`). Именно на него ссылается `ObjectTemplate.ObjectSelectionSetId`.
- **`SelectionSets`** (`TArray<FB1SelectionSet>`) — главное поле: список групп «состояние → набор
  пунктов». Разобрано ниже.
- **`IsDisplayObjectString`** (`bool`) — показывать ли отображаемые имена объектов в UI выбора.
- **`bShowInteractionOnGroupClick`** (`bool`) — если `true`, при клике по группе сразу показываются
  доступные интеракции.
- **`Comment`** (`string`) — служебная заметка разработчика (в данных обычно пустая).

## Вложенная структура: `FB1SelectionSet`

Каждый элемент массива `SelectionSets` — это одна группа состояния. У неё всего три поля, но именно
они — сердце реестра:

- **`StateGroupId`** (`FName`) — в каком **состоянии** предмета действует этот набор. Пункты меню
  привязаны не к предмету вообще, а к его текущему состоянию: рабочий унитаз и сломанный предлагают
  разные действия.
- **`SelectionType`** (`EB1SelectionType`) — тип поведения выбора. Возможные значения:
  - `Normal` — обычный выбор (в подавляющем большинстве строк именно он);
  - `Remote` — выбор через удалённое управление;
  - `Door` — выбор, относящийся к двери.
- **`InteractionBagList`** (`TArray<FName>`) — собственно список ID пунктов меню (внешние ключи на
  `InteractionBag.Id`). **Вот сюда дописывается новый пункт**, чтобы он появился в игре.

### Про `StateGroupId` — какие состояния бывают

Официальная документация не перечисляет значения (это открытый `FName`), но по живым данным видно
устойчивый набор:

| StateGroupId | Что означает | У кого встречается |
|---|---|---|
| `Normal` | обычное состояние | у всех |
| `Use` | предметом пользуются | `Toilet01` |
| `OnOff` | включён/выключен | `Computer01` |
| `Broken` | сломан | `Toilet01`, `Computer01` |
| `ElectricityCutoff` | отключено электричество | `Computer01` |
| `Garbage` | превратился в мусор | `Computer01` |
| `MakeLove` | особый сценарий | `Bed01` |

То есть у сложной техники состояний больше, и в каждом — свой список действий. Один и тот же
`InteractionBag` может встречаться сразу в нескольких состояниях (например `Computer_Use` есть и в
`Normal`, и в `OnOff`), а может быть уникальным для одного (`Object_PC_Repair_Home` — только в
`Broken`).

## Насколько разными бывают реестры

| Предмет | Состояний | Размер списков | Особенности |
|---|---|---|---|
| `Toilet01` | 3 (`Normal`/`Use`/`Broken`) | единицы пунктов | простая бытовая мебель |
| `Bed01` | 2 (`Normal`/`MakeLove`) | ~13 в `Normal`, **0** в `MakeLove` | пустой список — валидное состояние-заглушка |
| `Computer01` | 5 | **~200** в `Normal`/`OnOff`, единицы в остальных | завязан на десятки систем: карьера, преступления, обучение, страхование, карма… |

Вывод: формат один и тот же, но «наполненность» отличается на порядки. У кровати `MakeLove` — пустой
`InteractionBagList`: состояние существует, но пунктов в нём нет. У компьютера `Normal` — это буквально
сотни действий, потому что он служит порталом почти во все игровые подсистемы.

## Зачем это знать модеру: сюда регистрируется новое действие

Всё, что нужно, чтобы **новый пункт меню появился в игре**, — это дописать его ID в нужный
`InteractionBagList`. Практический путь:

1. Найти реестр предмета: `Object.<Item>.ObjectTemplateId` → `ObjectTemplate.<...>.ObjectSelectionSetId`
   → `ObjectSelectionSet.<Id>`.
2. Выбрать группу состояния, в которой пункт должен быть виден (чаще всего `Use` или `Normal`).
3. В её `InteractionBagList` **дописать** ID своего `InteractionBag` — не заменяя существующие.

Если ID не зарегистрирован здесь, сама строка `InteractionBag` может существовать, но игрок её не
увидит — реестр её просто не «покажет».

```{caution}
`SelectionSets` и вложенный `InteractionBagList` — это массивы/структуры, а их MCP `set_data`
стрингифицирует и ломает (см. {doc}`13-gotchas`). Регистрировать новый пункт нужно **вручную в
редакторе ModKit**. MCP удобен, чтобы найти реестр и свериться со списком.
```

## Каталог против меню: почему список длиннее, чем видно в игре

Частый сюрприз: в `InteractionBagList` какого-нибудь состояния лежат десятки ID, а в игре у предмета
всего два-три пункта. Так и задумано. **`InteractionBagList` — это исчерпывающий каталог всего, что
предмет умеет в этом состоянии на все случаи жизни, а меню игрока — отфильтрованный из него срез под
текущую ситуацию персонажа.**

Живой пример: у `ObjectSelectionSet.Toilet01` в группе `Normal` — **14** пунктов, а обычный взрослый
зой видит только `Toilet01_Use` и `Toilet01_Use_Flush`. Остальные 12 отсекаются. Почему — ниже.

### Четыре механизма, которые решают, показать пункт или нет

**1. Условия (`Condition_*`) — главный фильтр.** У самого `InteractionBag` есть три поля-«ворот», и
каждое ссылается на строки условий:

- `InteractionVisibleConditionIds` — видно ли пункт в меню;
- `InteractionEnableConditionIds` — активен ли он (можно ли кликнуть);
- `EnqueueConditionIds` — можно ли поставить в очередь.

**Пустой список = никаких ограничений, доступно всегда.** Именно поэтому видны ровно `Toilet01_Use` и
`Toilet01_Use_Flush` — у них эти поля пусты. А, например, `Toilet01_Use_MorningSickness` заперт на
условие `IsPregnancy_MorningSickness` и всплывёт только при токсикозе.

Сама строка условия — это маленькая логическая проверка. Вот реальный `Self_Uneasy_Lv2`, из-за
которого «срочные» варианты использования прячутся, пока нужда не станет острой:

```{jsonblock} Condition_Interaction.Self_Uneasy_Lv2
{
  "Id": "Self_Uneasy_Lv2",
  "AndConditionsList": [{ "Conditions": [
    { "BaseObject": "Self", "Command": "EmotionLevel", "CompareType": "Equal", "S1": "Uneasy", "F1": 2 }
  ]}]
}
```

Читается как «у самого персонажа уровень эмоции `Uneasy` равен 2». Пока это ложно — пункт скрыт.
(Подробно про движок условий — в {doc}`05-interaction-pipeline`.)

**2. Автономия vs клик игрока.** Часть пунктов вообще не предназначена для меню — их выбирает система
автономного поведения NPC. Опознаются по тегам частоты `IR_Freq_*` и по `AutonomyGroupTag`. Например
`Object_Toilet_Cry_Self` (тег `IR_Freq_VeryRare`) — это «зой иногда сам сходит поплакать в туалет», а
не пункт, который жмёт игрок.

**3. Возраст (`AgingConditionFlags`).** Битовая маска допустимых возрастных групп прямо в числе
(`508` — почти все возрасты, `504` — чуть уже). Не подходит возраст — пункт недоступен.

**4. Явное скрытие (`DisplayInfo.bHideUI`).** Флаг «не показывать в UI», даже если условия прошли —
для чисто служебных/скриптовых пунктов.

### Как это применять в своём моде

Когда добавляешь новый пункт, ты сам решаешь его видимость через эти же поля:

| Хочешь, чтобы действие… | Что сделать |
|---|---|
| было доступно **всегда** | оставить `InteractionVisibleConditionIds` / `InteractionEnableConditionIds` **пустыми** (как у `Toilet01_Use_Flush`) |
| появлялось **только в ситуации X** | вписать в `…VisibleConditionIds` ID подходящего `Condition_*` (переиспользуй существующий или заведи свой) |
| было видно, но иногда **некликабельно** | условие в `EnableConditionIds`, но не в `VisibleConditionIds` (пункт виден серым) |
| выбиралось **только автономией** NPC, не игроком | не давать «кликабельного» контекста, повесить `AutonomyGroupTag` и частотный тег `IR_Freq_*` |
| работало **только для нужного возраста** | выставить `AgingConditionFlags` |

```{tip}
Смысл разделения: `ObjectSelectionSet` отвечает за вопрос «*может ли* этот пункт вообще относиться к
предмету в этом состоянии», а поля условий внутри `InteractionBag` — за «*показать ли* его прямо
сейчас». Регистрация в `InteractionBagList` — необходимое, но не достаточное условие: без записи в
реестр пункт не появится никогда, но и с записью он покажется, только если пройдёт свои условия.
```
