# Известные проблемы и ловушки

Вещи, которые выглядят как обычный вызов, но ведут себя не так, как ожидаешь.

## set_data / set_data_batch стрингифицируют массивы и структуры

Подтверждено экспериментально: при передаче `TArray`/`Struct` в качестве `value` инструменты
`set_data`/`set_data_batch` иногда записывают их как ОДНУ СТРОКУ (JSON, завёрнутый в кавычки), а не
как настоящий массив/объект — даже для одноэлементного массива, и даже при перезаписи уже
существующего, ранее корректного поля (реальный кейс:
`InteractionBag.Toilet01_Use.FinishScriptIds` был нормальным массивом до правки и стал строкой
после). Скаляры (string/number/bool) пишутся нормально — проблема именно с массивами/структурами.

Симптом: `get_row` после записи показывает поле как `"[\"a\", \"b\"]"` (строка с экранированными
кавычками) вместо `["a", "b"]`; иногда параллельно ключ `Id` новой строки съезжает в нижний регистр
(`id`) — похоже на создание нетипизированной «заглушки» вместо настоящей типизированной строки
таблицы.

```{tip}
**Что делать.** Для полей типа массив/структура (`FinishScriptIds`, `Scripts`,
`InteractionSequence` и т.п.) — править вручную в самом ModKit-редакторе, не через MCP. Для
скаляров (ID валюты, числа, bool, enum-имя) — MCP работает надёжно, весь опыт этой сессии это
подтверждает.
```

## Регистр имён полей (PascalCase vs camelCase) — на практике не критичен

При ручном создании строки через сам ModKit-редактор один раз получилось так, что все поля
записались в `camelCase` (`id, scripts, ifType, baseObject, s1, f1, prob`) вместо привычного
`PascalCase` (`Id, Scripts, IfType, BaseObject, S1, F1, Prob`), которым записаны все остальные
строки таблицы. Выглядело как гарантированный баг — но **в игре скрипт сработал корректно** (валюта
начислилась). Похоже, импорт данных на стороне движка регистронезависим для этих полей. Тем не
менее для консистентности лучше придерживаться PascalCase, как в базовых данных игры — просто
теперь известно, что это не жёсткое требование.

## book_class='Script' через add_workspace — не заработал

Попытка «начеканить» новую скрипт-строку через `add_workspace(book_class='Script',
mode='Duplicate')` провалилась: `list_categories(book_class='Script')` и
`list_books(book_class='Script')` возвращают пустой список, так что валидный `book` для этого
класса подобрать не удалось. Прямой `set_data` на несуществующий ID тоже не создаёт нормальную
типизированную строку (см. проблему выше). Надёжный способ добавления новых `Script_Interaction`-строк
через MCP пока не найден — работает только через UI редактора (см.
{doc}`../case-studies/toilet-money`).

## Открытый баг: пропадают модели предметов при упаковке нетронутого воркспейса

BUILD-воркспейс, созданный (`add_workspace`) но ни разу не тронутый (`set_mesh`/`assign_material`/
`compile_bp` не вызывались), после `package` и загрузки мода в игре приводил к исчезновению 3D-модели
предмета — при этом интеракция и игровая логика продолжали работать нормально. Отключение мода
возвращало модели. Рабочая гипотеза: нетронутый Override при паковке экспортирует blueprint без
ссылки на меш.
